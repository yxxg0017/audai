# Audai 设计文档

## 一、产品愿景与设计原则

### 我们要解决什么问题

Audai 是一个 AI 视觉对话助手。核心场景：面对摄像头，用自然语言询问面前有什么、读一下画面里的文字，AI 结合实时画面用语音回答。

与传统的先拍照再提问不同，Audai 的目标是像视频通话一样自然——说话、AI 听到、AI 看到画面、AI 说话回答，整个过程无需点击或切换界面。

### 核心设计取舍

| 取舍 | 选择 | 理由 |
|---|---|---|
| 视频上传策略 | 不上传连续视频流，按需抽帧 | 连续视频流带宽和 token 成本不可控 |
| 语音传输模式 | 按一句话一个 turn 上传，非持续流式 | 简化服务端逻辑，降低静音期间的无效传输 |
| STT/TTS 部署 | 本地优先（whisper.cpp + say/Piper），云端兜底 | 零成本满足 demo，云端 API 作为备选 |
| LLM 推理 | 始终云端（OpenAI 兼容 API） | 本地 LLM 对硬件要求高，不具备普适性 |
| 架构风格 | 渐进增强：先 HTTP+SSE 跑通闭环，再优化延迟 | 降低初期复杂度，确保有限时间内可演示 |
| 移动端支持 | 第一公民，与桌面端同优先级 | 移动端摄像头是核心视觉输入场景 |

### 技术选型理由

- Next.js App Router：API Route 可同源代理本地服务，解决手机 HTTPS 混合内容问题
- Web Audio API + MediaRecorder：浏览器原生能力，无需额外 SDK
- whisper.cpp：本地 STT 零成本、离线可用、中文效果可接受
- macOS say / Piper：本地 TTS 零成本、即时可用
- canvas 抽帧：前端压缩图片至可控体积，再上传视觉 API
- SSE：单向流式推送 LLM delta 和 TTS 音频，比 WebSocket 更简单调试

## 二、初始用户故事设计

以下用户故事从使用场景角度设计。优先级：高（核心闭环必需）、中（提升体验和可观测性）、低（锦上添花）。

| 用户故事 | 优先级 | 设计理由 |
|---|---|---|
| 首次进入时填写 API Key、Base URL 和模型配置，保存后下次打开无需重复输入 | 高 | demo 需要灵活切换 API 端点；localStorage 是最轻量的持久化方案 |
| 使用过程中随时打开设置面板修改或清除配置 | 中 | 现场演示可能临时调整模型或 API 地址 |
| 点击开始后，浏览器请求摄像头和麦克风权限，授权后看到本地预览画面 | 高 | 所有后续功能的前置条件；预览画面确认 AI 的视野范围 |
| 实时看到麦克风输入电平变化 | 中 | 确认麦克风工作正常，避免说话后没反应却不知道原因 |
| 手机端在前置和后置摄像头之间切换 | 中 | 自拍用前置，展示桌面或环境用后置，覆盖两种典型场景 |
| 切换摄像头失败时自动恢复到之前可用的摄像头，并给出提示 | 低 | 部分移动浏览器对摄像头切换支持不稳定，降级比崩溃好 |
| 点击开始语音后直接说话，系统自动判断一句话的起止，无需按住按钮 | 高 | 自然对话的基本体验——人不习惯对着机器按住按钮说话 |
| 说出的话被转写成文字并实时显示在页面上 | 高 | 确认 AI 听懂了，也是链路可观测性的核心 |
| AI 逐句生成文字回复，并尽快转成语音播放 | 高 | 不需要等完整回答才听到声音，降低体感延迟 |
| AI 正在说话时直接插话打断，AI 停止当前回复开始听新问题 | 高 | 模拟真实对话中抢话的自然行为，避免干等 AI 讲完一大段 |
| 在设置中选择不同的本地 TTS 声线 | 中 | 不同演示场景可能偏好不同声音风格 |
| 点击分析画面，AI 描述当前摄像头中看到的内容 | 高 | 验证视觉链路独立的拍照问 AI 能力 |
| 语音对话中问面前有什么、看一下画面等视觉问题，系统自动抓取当前画面并结合内容回答 | 高 | 视觉与语音融合的核心闭环——不用先点按钮再说话 |
| 一分钟内连续追问类似视觉问题时，系统复用最近的画面分析结果，不重复调用 | 中 | 连续追问时的响应速度和 API 成本都依赖这个优化 |
| 问不同类型视觉问题时，AI 按对应意图给出针对性回答（物体识别、文字读取、场景描述、位置判断） | 中 | 不同视觉问题的答案侧重点不同，物体识别需要指出依据，文字读取需要精确 |
| 看到语音对话各环节的耗时（转写、思考、首句语音） | 中 | 演示时解释为什么快或慢，需要数据支撑 |
| 看到视觉分析的成本统计（调用了多少次、缓存命中了多少次、每次上传的图片多大） | 中 | 证明成本控制策略真实有效 |
| 在手机上通过局域网打开应用，使用全部功能 | 中 | 手机是移动摄像头的自然载体——拿着手机拍桌面比抱着笔记本方便 |

## 三、初始成本控制策略设计

在设计阶段，系统性地考虑了四个维度的成本控制：

### 视觉成本

| 策略 | 预期效果 | 实施复杂度 |
|---|---|---|
| 不上传连续视频流，只按需上传单帧 | 避免视频流的持续 token 消耗 | 低 |
| 前端 canvas 压缩图片（缩放 + 质量降低 + WebP 优先） | 单张图片体积控制在 100KB 以内 | 低 |
| 视觉 API 使用低细节模式（detail: low） | 大幅降低视觉 token 消耗 | 低 |
| 视觉摘要缓存（60 秒窗口） | 静止画面或连续追问避免重复调用 | 中 |
| 语音关键词触发视觉 tool，非视觉问题不上传图片 | 过滤掉不需要视觉的对话 | 中 |
| 图像相似度检测（基于像素差异） | 比时间窗口更精确地判断画面变化 | 高 |
| 图片 data URL 大小限制 | 防止异常大图消耗过多 token | 低 |

### 语音成本

| 策略 | 预期效果 | 实施复杂度 |
|---|---|---|
| VAD 按一句话提交音频，静音不触发请求 | 避免空白音频浪费 STT 调用 | 中 |
| 本地 STT（whisper.cpp），零云端 STT 成本 | 节省 OpenAI Whisper API 费用 | 高 |
| 本地 TTS（macOS say / Piper），零云端 TTS 成本 | 节省 OpenAI TTS API 费用 | 高 |
| TTS 按句分段合成，尽快播放首句 | 减少等待时间，同时避免一次性合成长文本 | 中 |

### 模型调用

| 策略 | 预期效果 | 实施复杂度 |
|---|---|---|
| 系统提示限制回复不超过 3 句话 | 控制每次对话的 token 消耗 | 低 |
| 视觉模型自动兜底（不支持图片的模型自动切换） | 避免无效调用浪费 token | 中 |
| 视觉问题合并到 LLM 多模态请求，不拆分两次调用 | 减少串行等待和额外 API 费用 | 中 |

### 基础设施

| 策略 | 预期效果 | 实施复杂度 |
|---|---|---|
| 对视觉 API 按客户端做请求限流（20 次/小时） | 防止误操作或滥用导致高额账单 | 低 |
| 对文本对话 API 按客户端做请求限流（40 次/小时） | 控制 LLM 调用频率 | 低 |
| 对 TTS 请求做限流（240 次/小时） | 防止极端高频 TTS 消耗 | 低 |
| 对 STT 请求做限流（180 次/小时） | 防止极端高频 STT 消耗 | 低 |
| 对 Realtime session 创建做限流（12 次/小时） | 防止频繁重建 WebRTC 连接 | 低 |
| 使用临时 client secret，前端不持有长期 API Key | 降低密钥泄露风险 | 中 |
| 前端展示成本面板（请求次数、缓存命中、图片体积） | 让评委直观看到成本控制效果 | 低 |
| 同源代理本地语音服务，避免暴露本地端口到公网 | 降低安全风险，同时解决移动端混合内容问题 | 中 |
| 生产级分布式限流（Redis / 托管 KV） | 多实例部署时共享限流状态 | 高 |

### 未实现/未采用项的原因

**图像相似度检测**

原始设计意图：通过比较连续两帧的像素差异，判断画面是否有实质变化，只有变化超过阈值才重新调用视觉 API，比单纯时间窗口缓存更精确。

未采用原因：
- 时间窗口缓存（60 秒摘要 + 15 秒压缩帧）在 demo 场景中已足够有效——静止画面在窗口内被重复分析的概率很低
- 图像相似度需要额外实现 perceptual hash 或直方图比较逻辑，引入新的代码复杂度
- 项目周期中边际收益不高，优先级低于手机端摄像头切换和同源代理等更影响演示效果的问题
- visual tool 的 15 秒压缩帧复用已经覆盖了连续追问场景，相似度检测主要解决的是更长时间跨度的静止画面优化

**生产级分布式限流**

原始设计意图：使用 Redis 或 Cloudflare KV 存储限流状态，使多个服务实例共享同一限流计数器。

未采用原因：
- 当前单实例部署（npm run dev 本地开发）中，内存 Map 限流完全满足需求
- 引入 Redis 需要额外运维（安装、配置、连接管理），超出 demo 范围
- 设计文档明确标注此限制，评委可理解这是 demo 而非生产系统
- 限流逻辑已抽象为独立模块（rate-limit.ts），后续替换存储后端改动量小

**全量本地多模态模型**

原始设计意图：将视觉理解也放在本地运行（如 LLaVA 等开源多模态模型），完全消除云端视觉 API 成本。

未采用原因：
- 本地多模态模型对 GPU 显存要求高（通常需 8GB 以上），不具备普适性
- 模型下载和部署流程复杂，与评委一键复现的目标冲突
- 云端视觉 API 在 demo 频次下成本可控（20 次/小时限流 + 缓存复用），优先级低于打通语音链路
- 即使采用本地模型，LLM 推理仍需云端——全链路本地化收益不完整

**高质量云端 TTS**

原始设计意图：使用 OpenAI TTS 或类似云端服务，获得更自然的语音合成效果。

未采用原因：
- 本地 TTS（macOS say / Piper）零成本且即时可用，满足 demo 语音可懂度要求
- 引入云端 TTS 增加 API 调用成本和额外网络延迟
- 本地 TTS 体现了端云协同的设计理念——能用本地的就用本地
- macOS say 的首句合成速度极快（通常小于 100ms），体感优于等待云端返回

## 五、核心架构说明

### 为什么主语音链路选择前端 VAD + HTTP POST + SSE

备选方案是 OpenAI Realtime/WebRTC 持续音频流。选择 HTTP + SSE 作为主链路的原因：

- HTTP POST 按 turn 提交音频，天然匹配一句话一次回答的对话模式
- SSE 比 WebSocket 更简单：单向推送，浏览器原生 fetch 流式读取即可
- 更容易插入中间处理环节——visual tool call 在 STT 完成后、LLM 开始前触发，有明确的事件边界
- 更容易展示链路状态——STT 完成、视觉等待、LLM 首 token、TTS 首音频等阶段都有独立事件
- 服务端可以用标准 Node.js HTTP 服务器实现，降低部署门槛
- OpenAI Realtime/WebRTC 仍保留为可选模式（voiceMode: realtime），供需要更低延迟的场景使用

### 为什么本地 STT/TTS 优先于云端

- whisper.cpp 中文转写效果在 large-v3-turbo 模型下可接受，适合 demo
- macOS say 合成速度极快（首句通常小于 100ms），降低了 TTS 环节的体感延迟
- 零 API 成本是最强的运营成本控制——STT 和 TTS 是每次对话都会调用的高频接口
- 云端 STT/TTS 仍通过 /api/stt 和 /api/tts 提供，作为本地服务不可用时的兜底
- 设置页支持自行选择 STT 提供方（浏览器静默/云端/本地）和 TTS 提供方（浏览器内置/本地）

### 视觉 tool 调用的设计选择

最初考虑两种方案：
- 方案 A：每次语音提问前都预先生成视觉摘要，注入 LLM 上下文
- 方案 B：语音转写后判断是否包含视觉意图，需要时才触发视觉 tool

选择了方案 B，理由：
- 避免非视觉问题浪费视觉 API 调用——只有真正需要看画面的问题才触发抓帧
- visual tool 的返回（图片 data URL）直接并入多模态 LLM 请求，LLM 同时看到图片和用户问题，回答质量高于先摘要再回答的两步串行
- 本地语音服务作为 tool 编排中心，内置 5 种 visual tool（画面描述、物体识别、文字读取、场景概括、位置判断），每种有独立的关键词匹配和提示词
- 前端只负责抓帧回传，职责清晰——视觉 tool 的触发逻辑在服务端，前端是执行层
- 方案 B 的延迟更可控：STT 完成后同步等待 tool 结果（最多 8 秒超时），再统一送入 LLM，而非先等一轮摘要再等一轮回答

### 同源代理的设计理由

手机通过 HTTPS 访问 Next.js 应用时，如果直接请求 http://127.0.0.1:8766 的本地语音服务，浏览器会因混合内容策略拦截请求。

解决方案：Next.js API Route（/api/local-voice/turn、/api/local-voice/tool-result、/api/local-voice/health）作为同源代理，将请求从浏览器同源转发到本地 Node.js 语音服务。

优势：
- 手机端无需额外配置证书或开放本地端口到公网
- 代理层可以统一添加错误提示（如本地服务未启动时引导用户运行 npm run voice:local）
- 桌面端和移动端使用相同的 API 路径，前端代码无需区分环境

## 六、已知限制与后续计划

| 限制 | 说明 | 后续方向 |
|---|---|---|
| 图像相似度检测 | 当前仅使用时间窗口缓存（60 秒摘要 + 15 秒帧），静止画面超时后会重新调用视觉 API | 可增加 perceptual hash 或直方图差异检测，在时间窗口基础上叠加画面变化判断 |
| 单实例内存限流 | 当前限流使用 Node.js 内存 Map，服务重启后计数清零，多实例不共享状态 | 生产部署应替换为 Redis 或 Upstash Redis |
| localStorage 密钥存储 | API Key 保存在浏览器 localStorage，适合本地 demo 但不适合生产 | 生产环境应改为服务端密钥托管，配置页仅用于选择环境而非输入密钥 |
| 本地 TTS 自然度 | macOS say 合成效果机械，Piper 依赖自行配置声线模型 | 可封装一键下载 Piper 声线模型，或提供更高质量的声线推荐 |
| 移动浏览器兼容 | 部分移动浏览器对摄像头切换支持不稳定，前置后置切换可能失败 | 持续跟踪 WebRTC 标准演进，增加更多 facingMode 尝试策略 |
| 视觉上下文延迟 | 当前视觉上下文注入依赖语音转写完成，极端情况下可能比模型第一句回答稍晚 | 可考虑在 VAD 检测到说话开始后即预判视觉意图并提前抓帧 |
| demo 视频链接 | 最终提交前需录制完整 demo 视频并补充链接 | 按 README 中的演示流程录制 |

### 浏览器验证

1. 打开页面，在配置页输入 API Key、Base URL 和模型配置并保存
2. 刷新页面，确认已保存配置并直接进入主界面
3. 通过顶部设置菜单修改或清除配置
4. 点击开始，确认摄像头画面和麦克风电平可见
5. 手机端点击摄像头切换按钮，确认前置后置切换正常，失败时能恢复
6. 点击分析画面，确认 AI 描述当前画面
7. 点击开始语音，说普通问题，确认 STT 转写和 AI 语音回复正常
8. 说面前有什么或看一下画面，确认触发视觉 tool 并结合画面回答
9. AI 播放语音时插话，确认回复被打断
10. 打开成本面板，确认视觉请求次数、缓存命中、图片体积等数据可见
