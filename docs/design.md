# 设计文档

## 1. 产品目标

Audai 是一个 AI 视觉对话助手。用户授权摄像头和麦克风后，可以直接用语音询问当前画面，系统会结合实时语音、摄像头抽帧和多模态视觉分析给出自然回应。

项目重点不是持续上传视频流，而是用浏览器完成本地音视频采集，再把“实时语音”和“按需视觉抽帧”组合起来：

- 语音链路默认使用浏览器语音流水线：SpeechRecognition 转写、文本模型生成回复、SpeechSynthesis 播放；Realtime WebRTC 作为可选高级模式保留。
- 视觉链路使用摄像头本地预览、canvas 抽帧压缩和多模态模型分析。
- 服务端 API Route 负责保护长期 API Key、创建临时 Realtime 凭据、调用视觉模型和执行请求限流。
- 浏览器入口配置页负责保存本地 demo 所需的 API Key、Base URL 和模型配置，便于用户测试不同服务地址。

## 4. 核心交互流程

1. 用户打开页面。
2. 如果浏览器没有保存配置，先填写 API Key 和 Base URL。
3. 用户点击“开始”，浏览器请求摄像头和麦克风权限。
4. 页面显示本地摄像头预览和麦克风电平。
5. 默认语音流水线模式下，用户点击“开始语音”，浏览器语音识别把语音转成文字。
6. 前端请求 `/api/chat` 获取文本回复，再用浏览器语音合成播放。
7. 如果切换到 Realtime 模式，用户点击“连接语音”，前端请求 `/api/realtime/session` 获取临时 client secret 并建立 WebRTC 连接。
8. 如果用户问题包含视觉意图，前端抽取当前摄像头画面并请求 `/api/vision`。
9. 前端将视觉摘要和用户问题注入语音流水线或 Realtime 会话。
10. AI 通过语音回答，页面同步显示转写和状态。
11. 如果用户在 AI 说话时插话，前端发送 `response.cancel`，停止当前回复。
12. 用户可通过顶部“设置”菜单修改或清除本地配置。
13. 用户可通过顶部菜单查看状态、日志和成本，不离开主屏也能管理关键面板。

## 5. 技术架构

```text
Browser
  ├─ localStorage 保存 demo 配置
  ├─ getUserMedia 摄像头/麦克风采集
  ├─ video 本地预览
  ├─ canvas 抽帧压缩
  ├─ SpeechRecognition 做语音转文字
  ├─ SpeechSynthesis 播放文字回复
  ├─ RTCPeerConnection 发送麦克风音频
  ├─ audio 播放 Realtime 远端音频
  └─ DataChannel 接收/发送 Realtime 事件

Next.js App Router
  ├─ /api/vision
  │   └─ 调用 OpenAI Responses API 分析压缩图片
  ├─ /api/chat
  │   └─ 调用 OpenAI Responses API 生成文本回复
  └─ /api/realtime/session
      └─ 调用 OpenAI Realtime client secrets API 创建临时凭据

OpenAI
  ├─ Realtime API：语音输入、语音输出、VAD、转写、打断
  └─ Responses API：摄像头抽帧图片理解
```

## 6. 视觉理解策略

- 不上传连续视频流。
- 用户点击“分析画面”时抽取当前帧。
- 用户语音命中视觉意图时自动抽帧。
- 抽帧使用 canvas 缩放，默认最大宽度 768px。
- 图片优先压缩为 WebP，必要时回退 JPEG。
- 视觉请求使用低细节输入，回答要求简洁具体。
- 视觉摘要缓存 60 秒，连续追问优先复用缓存。

## 7. 语音交互策略

- 默认模式是语音流水线。
- STT 使用浏览器 SpeechRecognition。
- VAD/说话结束由浏览器语音识别事件承担。
- LLM 回复通过 `/api/chat` 调用文本模型。
- TTS 使用浏览器 SpeechSynthesis。
- 如果中转站不支持 Realtime，仍可使用语音流水线完成语音问答。
- Realtime 模式保留在设置菜单中，适合支持 OpenAI Realtime API 的服务。
- 浏览器用 WebRTC 直接传输麦克风 audio track。
- 服务端只创建临时 client secret，不接触浏览器音频流。
- Realtime session 启用服务端 VAD。
- Realtime session 启用中文语音转写。
- data channel 用于同步用户说话、模型思考、AI 回复和错误状态。
- AI 回复过程中用户插话时，前端发送 `response.cancel`。
- AI 回复默认不超过 3 句话，降低延迟和 token 成本。

## 8. 想到的成本控制技巧

- 不持续上传视频，只在需要时抽帧。
- 降低图片分辨率和质量。
- 使用低细节视觉输入。
- 缓存最近视觉摘要，避免静止画面重复分析。
- 使用 VAD，避免静音时持续生成响应。
- 使用临时 Realtime 凭据，避免前端暴露长期密钥。
- 对高成本 API 做限流。
- 控制回答长度，减少无效输出。
- 在 UI 中显示请求次数和图片体积，方便调试成本。

## 9. 实际采用的成本控制技巧

- 摄像头画面只在本地预览，不持续上传。
- 视觉分析只上传压缩后的单帧图片。
- 抽帧图片最大宽度 768px。
- 图片 data URL 长度限制为 2,000,000。
- 用户问题长度限制为 500 字符。
- 视觉模型使用 `detail: low`。
- 视觉摘要缓存 60 秒。
- `/api/vision` 每客户端每小时最多 20 次请求。
- `/api/realtime/session` 每客户端每小时最多 12 次请求。
- Realtime 常规回答限制在 3 句话以内。
- 前端成本面板展示视觉请求次数、缓存命中和最近图片体积。

## 10. 已知限制

- 真实语音和视觉效果需要配置有效 `OPENAI_API_KEY` 后在浏览器手动验证。
- 视觉上下文注入依赖语音转写完成，极端情况下可能比模型第一句回答稍晚。
- 当前视觉缓存基于时间窗口，没有实现图像相似度检测。
- 当前限流使用单实例内存计数，生产多实例部署时应替换为 Redis 或托管 KV。
- demo 视频链接需要最终录制后补到 README。
- API Key 如果通过配置页输入，会保存到浏览器 localStorage；这适合本地 demo，不适合作为生产级密钥托管方案。

## 11. 验证方式

自动/命令验证：

```bash
npm run lint
npm run typecheck
npm run build
npm audit
npm run dev -- --port 3000
curl -I http://localhost:3000
```

浏览器手动验证：

1. 打开应用，在配置页输入 API Key、Base URL。
2. 刷新页面，确认已保存配置并直接进入主界面。
3. 通过顶部“设置”菜单修改或清除配置。
4. 启动应用并授权摄像头、麦克风。
5. 点击“分析画面”，确认能看到视觉结果。
6. 默认语音流水线模式下，点击“开始语音”，确认浏览器语音识别、文本回复和语音播放可用。
7. 通过设置菜单切换到 Realtime 模式，确认 Realtime 语音连接成功。
8. 说“我面前有什么”，确认视觉上下文被注入并得到回答。
9. 在 AI 回复时继续说话，确认当前回复可被打断。
10. 观察成本面板中的视觉请求次数、缓存命中和图片体积。
