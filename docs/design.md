# Audai 设计文档

## 1. 项目概述

Audai 是一个面向议题实战的 AI 视觉对话助手。用户授权摄像头和麦克风后，可以通过语音询问当前画面，例如“我面前有什么”“读一下画面里的文字”，系统会采集当前语音 turn、按需抽取摄像头画面，并通过本地语音服务与多模态模型返回文字和语音回复。

项目的核心设计取舍是：不持续上传视频流，而是让浏览器负责本地音视频采集，只在用户明确触发视觉意图时上传压缩单帧；语音链路默认使用 HTTP POST 上传音频 turn，并通过 SSE 返回 STT、LLM、TTS、视觉 tool 和耗时指标。OpenAI Realtime/WebRTC 作为可选高级模式保留在设置中。

## 4. 当前系统架构

```text
Browser
  ├─ localStorage 保存 demo 配置
  ├─ getUserMedia 采集摄像头/麦克风
  ├─ video 本地预览，可切换前置/后置摄像头
  ├─ Web Audio RMS VAD 检测用户说话
  ├─ MediaRecorder 录制当前语音 turn
  ├─ canvas 抽取并压缩当前摄像头帧
  └─ fetch 同源 Next API，读取 SSE 事件

Next.js App Router
  ├─ /api/local-voice/turn
  │   └─ 代理到本机 Node 语音服务 /voice/turn
  ├─ /api/local-voice/tool-result
  │   └─ 代理视觉 tool 回传图片
  ├─ /api/local-voice/health
  │   └─ 代理本机语音服务健康检查
  ├─ /api/vision
  │   └─ 手动画面分析
  ├─ /api/chat
  │   └─ 文本流式回复
  └─ /api/realtime/session
      └─ 可选 Realtime 临时凭据

Local Node Voice Service
  ├─ whisper.cpp 本地 STT
  ├─ 视觉关键词映射 tool
  ├─ OpenAI-compatible Chat Completions 流式 LLM
  ├─ macOS say / Piper 本地 TTS
  └─ SSE 返回 stt.final、tool.call、tool.result、metric、llm.delta、tts.audio、done
```

## 5. 关键实现方案

### 5.1 语音流水线

语音主链路采用“前端 VAD + HTTP POST + SSE”的方式，而不是持续 WebSocket 音频流：

1. 前端持续监听麦克风 RMS。
2. 检测到用户说话后开始记录当前 turn，并带上约 500ms 预录缓冲。
3. 连续静音达到阈值后停止录音。
4. 前端把 WebM/Opus 音频通过 `POST /api/local-voice/turn` 上传。
5. Next 同源代理把请求转发到本地 Node 语音服务。
6. Node 服务完成 STT、视觉 tool、LLM 和 TTS 编排。
7. 前端通过 SSE 逐步接收转写、模型文本、音频和指标。

### 5.2 视觉对话

视觉能力分为手动分析和语音触发两种：

- 手动分析：点击“分析画面”，前端抽帧并请求 `/api/vision`。
- 语音触发：当 STT 文本包含“画面、看到、面前、这是什么、读一下、在哪里”等关键词，Node 服务发出 `tool.call`，前端自动抓帧回传。

早期方案是先调用一次视觉模型生成摘要，再调用文本模型回答。最终实现改为把图片直接并入本轮多模态流式回答，减少一次模型串行调用，降低图片问答延迟。

### 5.3 手机端支持

手机端重点解决两类问题：

- 摄像头方向：提供前置/后置切换；切换前释放旧摄像头轨道，降低移动浏览器因设备占用导致的失败。
- 本地语音服务访问：手机 HTTPS 页面不直接请求 `http://127.0.0.1:8766`，统一请求同源 `/api/local-voice/turn`，由 Next 服务端转发到 Mac 本机语音服务。

### 5.4 可观测性

为了证明链路真实运行，页面展示：

- 摄像头预览和麦克风电平。
- STT 状态、turnId、音频大小和 mime type。
- 最近用户转写和 AI 流式回复。
- 视觉 tool 结果和视觉缓存状态。
- STT、视觉等待、LLM 首 token、TTS 首音频、总耗时等指标。

## 6. 想到的运营成本控制技巧

项目设计阶段考虑过以下成本控制方式：

1. 不持续上传视频流，只按需上传单帧图片。
2. 前端压缩图片尺寸和质量，减少视觉输入 token 与网络开销。
3. 使用低细节图片输入。
4. 视觉摘要或压缩帧缓存，避免静止画面连续重复分析。
5. 根据用户语音关键词触发视觉 tool，非视觉问题不上传图片。
6. 使用 VAD，只在用户完成一句话后提交音频，避免静音期间持续请求。
7. 本地 STT/TTS 优先，减少云端语音 API 成本。
8. 控制 LLM 回复长度，避免长回复带来 token 和 TTS 成本。
9. 对高成本 API 做限流，例如视觉分析和 Realtime session。
10. 使用 Realtime 临时凭据，避免长期密钥暴露导致滥用成本。
11. 在前端展示成本指标，让 demo 时能看到请求次数、缓存命中和图片体积。
12. 对模型做自动选择和兜底，避免误用不支持视觉的模型导致重复失败调用。
13. 同源代理本地服务，减少移动端额外部署或公网暴露本地服务的成本。
14. 对 TTS 音频按句生成，尽快播放首句，同时避免一次性合成长文本。

## 7. 实际采用的成本控制技巧

最终实际落地的策略如下：

| 策略 | 是否采用 | 实现说明 |
| --- | --- | --- |
| 不上传连续视频流 | 已采用 | 摄像头只在浏览器本地预览，视觉问题触发时才上传单帧。 |
| 图片压缩 | 已采用 | `canvas` 抽帧后缩放并压缩，默认较小宽度和质量，优先 WebP，必要时 JPEG。 |
| 低细节视觉输入 | 已采用 | 视觉请求使用 `image_url.detail = "low"`。 |
| 图片大小限制 | 已采用 | `/api/vision` 限制 data URL 最大长度，避免异常大图。 |
| 视觉缓存 | 已采用 | 视觉上下文默认缓存 60 秒，视觉 tool 复用 15 秒内最近压缩帧。 |
| 关键词触发视觉 tool | 已采用 | 非视觉问题不会自动上传图片；视觉关键词才触发抓帧。 |
| VAD 语音 turn | 已采用 | 前端按一句话上传音频，静音和短噪声不应触发有效请求。 |
| 本地 STT | 已采用 | 本地 Node 服务接入 whisper.cpp，默认使用 `large-v3-turbo`，模型文件不提交仓库。 |
| 本地 TTS | 已采用 | 默认 macOS `say`，可配置 Piper 本地声线。 |
| 短回复约束 | 已采用 | 系统提示要求默认不超过 3 句话，多模态回答限制短输出。 |
| 请求限流 | 已采用 | `/api/vision` 和 `/api/realtime/session` 按客户端指纹限流。 |
| 成本面板 | 已采用 | 前端展示视觉请求次数、缓存命中、最近图片体积和缓存窗口。 |
| 阶段耗时指标 | 已采用 | SSE `metric` 返回 STT、视觉等待、首 token、首音频、总耗时，便于定位高成本/高延迟环节。 |
| 模型兜底 | 已采用 | 视觉模型不明显支持图片时，兜底到 `gpt-4o` 或 `[按次]gpt-4o`。 |
| 同源代理 | 已采用 | 手机端通过 Next API 访问本机语音服务，避免单独暴露 8766 端口到公网。 |

未采用或暂缓的策略：

- 图像相似度检测：可以进一步降低静止画面的重复分析，但 3 天实战中优先用时间窗口缓存。
- 统一云端高表现 TTS：效果更自然，但会增加 API 成本；当前优先本地 TTS。
- 生产级分布式限流：当前是单实例内存限流，足够 demo；生产应使用 Redis 或托管 KV。
- 全量本地多模态模型：可降低云成本，但本地硬件和部署复杂度不适合当前 3 天项目周期。

## 8. 验收与复现方式

命令验证：

```bash
npm run lint
npm run typecheck
npm run build
node --check scripts/local_voice_server.mjs
```

本地运行：

```bash
npm install
bash scripts/setup_local_voice.sh
npm run voice:local
npm run dev
```

浏览器验证：

1. 打开页面并保存 API Key、Base URL 和模型配置。
2. 点击“开始”，确认摄像头画面和麦克风电平可见。
3. 手机端点击“后置摄像头”，确认画面切换或失败后能恢复原摄像头。
4. 点击“分析画面”，确认 AI 可以描述当前画面。
5. 点击“开始语音”，说普通问题，确认 STT 和 AI 回复展示。
6. 说“看一下画面里有什么”，确认自动触发视觉 tool 并结合画面回答。
7. 在 AI 播放时插话，确认当前回复被停止。
8. 打开成本菜单，确认视觉请求次数、缓存命中和图片体积可见。

## 9. 当前限制与后续计划

- demo 视频链接仍需录制后补充到 README。
- 浏览器 localStorage 保存 API Key 只适合本地 demo，生产环境应改为服务端密钥托管。
- 本地 TTS 的自然度仍依赖用户配置的 Piper 声线；后续可封装一键下载和声线管理。
- 视觉缓存目前基于时间窗口，后续可增加图像差异检测。
- 当前限流为单实例内存实现，生产部署应替换为共享存储。
